## 推奨対応 (P1) — 上位3件

**[P1-1] タスク5・6完了後のゲート条件の追加**
- **出典**: Approach Reviewer [P1-2]
- **内容**: タスク5（旧テスト削除）とタスク6（テスト移設）の完了後にゲート条件が設定されていない。旧テスト削除後に移設テストを含む全テストが PASS することを確認するゲートがないと、テスト移設の正当性が未検証のまま後続タスクに進む可能性がある。ゲート条件の追加による品質確認の強化は対応効果が高い。
- **修正方針**: 9章タスク一覧のタスク6に「ビルド成功・全テスト PASS」のゲート条件を追加する。併せてゲート条件と切り戻し方針セクションに「タスク6完了後: ビルド成功・全テスト PASS を確認する。移設テストが失敗した場合は DI 構成またはテストの移設内容を修正する」を追記する。

**[P1-2] ON/OFF 比較系テストの depth=5 での不一致時の対応方針の具体化**
- **出典**: Technical Quality Reviewer [P1-1]
- **内容**: depth 削減に伴うアサーション更新方針において、ON/OFF 比較系テストで BestMove が不一致となった場合のテスト設計変更方針が記載されていない。「実測値に更新する」だけでは、テストの検証意図（ON/OFF 同値性）自体が成立しなくなるケースへの対応が不十分である。
- **修正方針**: 5.6 節の ON/OFF 比較系に「depth=5 で BestMove が一致しない場合は、テストの検証観点を『両方が合法手であること』に変更する。値の一致を確認するテストは depth=5 で一致する別の局面で再構成する」という方針を追記する。

**[P1-3] `[DoNotParallelize]` 対象テストクラスの将来的な網羅性担保**
- **出典**: Security & Risk Reviewer [P1-1]
- **内容**: 現在5つのテストクラスを `[DoNotParallelize]` 対象として列挙しているが、将来 TT を使用するテストクラスが追加された場合に付与漏れが発生するリスクがある。
- **修正方針**: 5.5 節の TT 共有に対する緩和策の末尾に「**注意:** TT を使用する新規テストクラスを追加する際は、必ず `[DoNotParallelize]` 属性を付与すること」という注意書きを追記する。
