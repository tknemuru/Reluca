**[P1-1] DI 切替後の動作等価性に関する検証基準が不明確**
- **対象セクション**: 8. テスト戦略 / 検証観点
- **内容**: 「UI 対局で正常に動作すること」が検証観点に含まれているが、「正常に動作する」の具体的な判定基準（応答時間の上限、手の妥当性の評価方法等）が定義されていない。`PvsSearchEngine` と `LegacySearchEngine` では探索結果が異なる可能性があると RFC 自身が認めており、何をもって「正常」とするかの基準が必要である。
- **修正の期待値**: 検証観点に「応答時間が N 秒以内」「合法手を返す」等の具体的な判定基準を追記する。

**[P1-2] 3つの変更の一括実施における切り戻し手順の欠如**
- **対象セクション**: 6. 代替案の検討 / 9. 実装・リリース計画
- **内容**: 案Bの短所として「変更範囲が広い」ことを認識しているが、問題発生時の切り戻し手順が記載されていない。DI 切替・旧コード削除・テスト変更を一括で行う場合、各段階でのチェックポイントと切り戻し方法を明示することで、リスクを緩和できる。
- **修正の期待値**: 実装計画にタスク間のチェックポイント（例: タスク2完了後に全テスト PASS を確認してからタスク3に進む）を追記する。

**[P2-1] テスト実行時間の目標値「30秒以下」の根拠**
- **対象セクション**: 3. 目的とスコープ / Goals
- **内容**: テスト実行時間の目標を「170秒から30秒以下に短縮」としているが、30秒という数値の根拠が示されていない。NegaMaxTest（34秒）の削除と並列実行の効果を概算すると妥当な範囲に見えるが、根拠を明記しておくと目標の妥当性が検証しやすくなる。
